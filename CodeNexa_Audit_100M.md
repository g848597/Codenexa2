## 1. Архитектура

**Общая оценка подхода**: для MVP выбор оправдан и осознан — авторы сами документируют компромиссы (см. `README_BACKEND.md`, `docs/LAUNCH_CHECKLIST.md`). Это редкость и большой плюс: обычно на аудит попадают проекты без единой строчки объяснения "почему так". Здесь видно, что решения принимались осознанно, а не случайно.


Структура:
```
app/web/          — весь бэкенд одним модулем FastAPI
  api/            — роутеры (auth, billing, investors, telegram)
  integrations/   — внешние API (CryptoBot, Stars)
  db.py, repo.py  — самодельный "ORM-lite" поверх psycopg2
webapp/           — фронтенд без сборки, чистые ES-модули
```

**Что хорошо:**
- Чёткое разделение `api/` (HTTP-слой) → `repo.py` (доступ к данным) → `db.py` (транспорт). Ни один SQL-запрос не собирается конкатенацией пользовательского ввода — только параметризованные запросы.
- Один процесс отдаёт и API, и статику — валидное решение для этой стадии, убирает целый класс CORS-проблем.
- Фронтенд: конфигурация (`config/*.js`) отделена от компонентов (`components/*.js`) и стилей — новый продукт в каталоге добавляется одним объектом в массив, без правки вёрстки. Это действительно масштабируемый паттерн для "конвейера ботов", который описан в доменной модели.

**Что не выдержит роста:**
- **`repo.py` — самодельная замена ORM без её преимуществ.** На 6 сущностях это терпимо. На 30+ (а роудмап явно предполагает рост каталога ботов, новые сущности для аналитики, рефералки, тарифов) это станет узким местом: нет миграционного инструмента (Alembic), нет валидации схемы на уровне Python, любое изменение таблицы — это ручной SQL в `db.py` плюс ручная миграция в `_INVESTOR_MIGRATIONS`. При росте команды это гарантированно приведёт к рассинхрону схемы между окружениями.
- **Нет слоя "сервисов"/use-cases.** Бизнес-логика размазана прямо по FastAPI-эндпоинтам (`auth.py` держит и HTTP-парсинг, и бизнес-правила rate-limit, и работу с OAuth state). Работает на 4 роутерах, но с ростом функций (см. раздел 9) эндпоинты будут раздуваться, и без выделения `services/` тестировать бизнес-логику отдельно от HTTP станет невозможно.
- **Нет тестов вообще** (см. раздел 8) — при текущей архитектуре (логика в роутерах) написать unit-тесты дорого, потребуется либо `TestClient`, либо рефакторинг в сервисы. Рекомендуем рефакторить до того, как логики станет вдвое больше — сейчас цена рефакторинга минимальна.
- **Single point of failure по дизайну**: один процесс = и сайт, и API, и раздача аплоадов. При деплое на Railway с одним инстансом это означает, что баг в любом из 4 роутеров (например, необработанное исключение в `investors.py` при аплоаде фото) может уронить весь сайт, включая посадочную/маркетинговую часть, которую видят инвесторы.
- **In-memory состояние там, где нужна персистентность**: `_attempts` (rate-limit) и `_pending_tokens` (OAuth-обмен) в `auth.py` живут в памяти процесса. Авторы сами это документируют как компромисс "для одного воркера" — но это значит, что горизонтальное масштабирование (>1 инстанс/воркер) сломает и rate-limit (обходится сменой инстанса за балансировщиком), и OAuth-логин через Google/Яндекс (пользователь получит "код недействителен", если запрос попадёт на другой воркер).
  **✅ ИСПРАВЛЕНО (раунд 3)**: rate-limit теперь опционально использует Redis (`app/web/cache.py`) — с `REDIS_URL` счётчик общий для всех воркеров, без него фолбэк на прежнюю память процесса, недоступность Redis не роняет запрос. `_pending_tokens` удалён полностью — OAuth-коды идут только через `oauth_links` в БД, что само по себе переживает рестарт и работает на нескольких воркерах без Redis.

**SOLID/DRY/KISS:**
- KISS соблюдён откровенно хорошо — нет избыточной абстракции ради абстракции.
- DRY нарушается по мелочи, но системно: функция `esc()`/`escAttr()` продублирована как минимум в 8 файлах фронтенда почти буквально. Это тот тип дублирования, который "выстрелит" в будущем — если понадобится усилить экранирование (например, добавить санитайзер для `href`), придётся синхронно чинить 8 мест.
- SRP: `deps.py` совмещает и извлечение пользователя из заголовка, и бизнес-правило "кто админ" — не критично, но с ростом ролей (см. раздел 9 — предлагаем полноценную ролевую модель) это надо будет развести.

**Дублирование:**

- Паттерн `google_callback`/`yandex_callback` в `auth.py` дублирует ~40 строк логики "найти/создать пользователя → touch_login → issue_token → сгенерировать one-time code → deep link" почти один в один. Один провайдер добавили — 90% кода скопировали. При добавлении 3-го/4-го OAuth-провайдера (Apple, VK — см. раздел 9) это стоит вынести в `_complete_oauth_login(provider_id_field, user_info, state)`.
- В `repo.py` шаблон `get_conn(); row = conn.execute(...).fetchone(); return row_to_dict(row)` повторяется 10+ раз — не критично, но можно свернуть в `_fetch_one(query, params)`.

**Именование**: в целом хорошее, по-русски откомментировано осмысленно (не "TODO fix", а объяснение решения). Отдельная похвала — комментарии в коде объясняют **почему**, а не **что** (редкость).

**Что переписать полностью:**
1. **Слой БД (`db.py`)** — самодельная "эмуляция sqlite3-интерфейса поверх psycopg2" (замена `?` на `%s` регуляркой, ручной `RETURNING id` через regex по SQL-тексту) — гениальный трюк для быстрой миграции с SQLite на Postgres, но это технический долг, который лучше закрыть сейчас, чем через год, когда запросов будет 10x больше. Regex-based rewriting SQL — источник трудноуловимых багов (например, если в тексте запроса встретится `?` внутри строкового литерала — редко, но возможно).
2. **`_attempts`/`_pending_tokens` in-memory** → Redis (см. DevOps). **✅ ИСПРАВЛЕНО (раунд 3)** — см. пометку в разделе 1.

**Где можно ускорить:**
- `is_session_valid()` в `repo.py` делает отдельный SELECT на каждый защищённый запрос (проверка сессии по JWT). При росте RPS это лишний round-trip к БД на **каждый** авторизованный вызов API. Кандидат на кэширование (Redis, TTL 60–120 сек) или на "ленивую" ревокацию (хранить в JWT версию сессии, проверять БД только раз в N минут).

## 3. Производительность

- **Нет пула соединений.** `db.py` открывает по одному psycopg2-соединению на поток (`threading.local`) и держит его вечно, переоткрывая только при обрыве. Для FastAPI, где sync-эндпоинты выполняются в `threadpool` (по умолчанию до 40 потоков), это означает **до 40 постоянно открытых соединений к Postgres с одного процесса**, и это число растёт с каждым доп. воркером. Supabase на бесплатном/стартовом тарифе Session Pooler ограничивает число соединений (обычно 15–60 в зависимости от плана) — при реальной нагрузке возможны `too many connections`. **Решение**: `psycopg2.pool.ThreadedConnectionPool` или переход на `asyncpg`/`SQLAlchemy async` с явным пулом.
  **✅ ИСПРАВЛЕНО (раунд 5)**: `db.py` переведён на `psycopg2.pool.ThreadedConnectionPool` (размер задаётся `DB_POOL_MIN`/`DB_POOL_MAX`, по умолчанию 1/10 на процесс). Соединение теперь берётся из пула и сразу возвращается на время одной единицы работы — одного запроса для `get_conn()`, одной транзакции для `tx()` — а не держится в `threading.local` до конца жизни потока. Намеренно НЕ использована схема "закэшировать в thread-local и вернуть в пул из middleware после запроса": sync-эндпоинты FastAPI выполняются в threadpool, а async-эндпоинты — в потоке event loop, поэтому начало и конец обработки одного HTTP-запроса не гарантированно проходят в одном и том же ОС-потоке — thread-local-привязка в этой схеме периодически "теряла" бы соединение и не возвращала его в пул. Acquire/release в пределах одного вызова `get_conn()`/`tx()` этой проблеме не подвержены. Синхронный psycopg2 внутри `async def`-эндпоинтов (следующий пункт) в этом раунде не тронут — это отдельная задача, не входившая в scope раунда 5.
- **Синхронный psycopg2 внутри `async def` эндпоинтов.** `google_callback`, `yandex_callback`, `cryptobot_webhook`, `telegram_webhook` — все объявлены `async def`, но внутри вызывают синхронные `repo.*` (блокирующий сетевой I/O к Postgres). Это **блокирует event loop** на время запроса к БД — при параллельных OAuth-логинах или вебхуках все остальные async-запросы (даже не связанные с БД) будут ждать. Решение: либо сделать эти эндпоинты `def` (тогда FastAPI сам уведёт их в threadpool), либо обернуть блокирующие вызовы в `run_in_threadpool`.
- **N+1 нет** — на текущем масштабе запросов (простые SELECT по 4-6 таблицам) явных N+1 не обнаружено, `investors` грузится одним запросом с сортировкой на уровне SQL (`ORDER BY sort_order` — правильно, не в Python).
- **Фронтенд без сборки**: 15+ отдельных JS-файлов и 15 CSS-файлов грузятся как отдельные HTTP-запросы без бандлинга/минификации/tree-shaking. На быстром соединении внутри Telegram WebView это терпимо на старте, но при росте каталога продуктов (роадмап предполагает "сегодня 4, завтра 20 ботов") это будет деградировать TTI (Time to Interactive), особенно в регионах с плохим мобильным интернетом — а именно там основная аудитория Telegram Mini Apps.
- **Изображения инвесторов** пережимаются на сервере до 1600px JPEG q88 — разумно, но нет генерации нескольких размеров (миниатюра для списка vs полный размер) и нет CDN — при росте списка инвесторов/партнёров это тоже отложенная проблема, не сейчас.

## 4. Безопасность

- Проверка `initData` Telegram реализована по спецификации, с `hmac.compare_digest` (защита от timing-атак) и проверкой возраста подписи — сделано образцово.
- Пароли — bcrypt через `passlib`, корректно.
- SQL — везде параметризованные запросы, инъекций не найдено.
- Загрузка изображений — редкий случай, когда все три классические дыры (poison-по-content-type, path traversal через имя файла, EXIF/полиглот-файлы) закрыты одновременно: реальная проверка через Pillow, генерация имени на сервере, перекодирование с уничтожением метаданных.
- 2FA (TOTP) реализована корректно, с `valid_window=1` (допуск дрейфа часов).
- XSS: пользовательский контент (имена инвесторов, описания) везде проходит через `esc()`/`escAttr()` перед вставкой в `innerHTML` — проверено выборочно по нескольким компонентам, паттерн соблюдается последовательно.

## 5. База данных

**Схема** (`db.py`, `SCHEMA`): 5 таблиц — `users`, `sessions`, `oauth_links`, `payments`, `investors`. Для MVP — адекватный, не избыточный набор.

**Что хорошо:**
- Внешние ключи с `ON DELETE CASCADE` — удаление пользователя корректно каскадно чистит сессии/платежи.
- Индексы есть на очевидных горячих путях: `idx_payments_user`, `idx_sessions_user`, `idx_investors_status_order`.
- "Ручные" идемпотентные миграции (`_INVESTOR_MIGRATIONS`) — работает, но не масштабируется (см. ниже).

**Проблемы:**
- **Нет индекса на `sessions.token_id`** несмотря на то что это `UNIQUE` (уникальный индекс создаётся автоматически, ОК) — но также нет индекса на `sessions.revoked` в паре с `token_id`, и **каждый авторизованный запрос** делает `SELECT revoked FROM sessions WHERE token_id = ?` — с ростом таблицы sessions (а она растёт с каждым логином и никогда не чистится) это станет самым частым запросом в системе. Нужен `TTL`/периодическая очистка старых revoked/просроченных сессий (partition или cron `DELETE ... WHERE created_at < NOW() - INTERVAL '90 days'`).
- **Нет ограничения уникальности на активные OAuth `state`** с TTL — `oauth_links` растёт бесконечно, старые `state`/`code` никогда не удаляются. Через год — таблица-мусорка, влияющая на планы запросов.
- **`amount` в `payments` — тип `TEXT`, а не `NUMERIC`.** Хранить деньги строкой — прямой путь к багам сортировки/агрегации ("100.00" vs "99.9" как строки сравниваются неверно) и уже проявляется как первопричина бага из п.0.3 (сумма гоняется как строка между `PLANS`, CryptoBot API и БД без единой денежной единицы/точности). **Рекомендация**: `NUMERIC(20,8)` + отдельная колонка `currency`, конвертация в USD-эквивалент на **сервере**, а не полагаться на клиента/провайдера.
  **✅ ИСПРАВЛЕНО (раунд 5)**: колонка переведена на `NUMERIC(20,8)` (идемпотентная миграция для существующих БД в `_run_migrations()`), добавлена единая денежная утилита `app/web/money.py` (`Decimal` везде внутри процесса, включая курс CryptoBot и сумму тарифа — раньше в `convert_usd_to_asset()` было `float`), наружу в JSON-ответах суммы отдаются форматированной строкой (`money.to_display()`), а не "голым" `Decimal`, чтобы сериализация не откатилась обратно к float-погрешности.
- **Никакой схемы для будущих сущностей** (тарифы у `PLANS` — просто Python-словарь в коде, не таблица). Значит, изменить цену тарифа = задеплоить код. Для финансового продукта, тем более под инвестора, тарифы обязаны жить в БД с историей изменений (аудит цен).
- **Резервное копирование**: полностью полагается на Supabase managed backups. Это нормально как база, но нет ни одного упоминания стратегии восстановления (RPO/RTO), нет проверенного процесса "восстановились из бэкапа и это сработало" — что для инвестора будет первым вопросом due diligence по data resilience.
- **Нормализация**: в целом 3NF соблюдена. Дублирование `investment_amount` (TEXT, для отображения) и `investment_amount_value` (DOUBLE PRECISION, для расчётов) — осознанный, а не случайный дубль (комментарии подтверждают), приемлемо.

## 6. UX/UI

*Дисклеймер: аудит кода UI, а не живого рендера в Telegram — часть выводов сделана по CSS/структуре компонентов, рекомендуем дополнить реальным usability-тестом на устройствах.*

**Что вызывает вопросы:**
- **Двойной прыжок при внешней OAuth-авторизации** ("нажал Google → открылся системный браузер → вошёл → нажал 'вернуться в Telegram' → бот открыл мини-апп заново"). Это технически вынужденное решение (WebView-ограничения Telegram), но с точки зрения конверсии — каждый лишний экран теряет 15-30% пользователей на мобильных воронках. Стоит явно измерять конверсию именно этого шага (см. раздел 9 — аналитика воронки).
- **Отсутствие единого системного индикатора загрузки/skeleton-состояний** — судя по коду, состояния "загрузка" реализованы как текст `"Загрузка аккаунта…"` в разных компонентах по-разному (где-то спиннер, где-то текст) — нет единого паттерна loading/skeleton, что визуально ощущается как "недособранный" продукт при первом впечатлении инвестора.
- **Пустые состояния (empty states)** — сделаны осознанно честно ("не показывать выдуманные метрики"), это правильная продуктовая философия, но с точки зрения первого впечатления страница с `metric.value: null` у всех продуктов выглядит как "продукт без трекшна" — для инвесторского лендинга это обоюдоострый меч: честность vs товарный вид. Рекомендация: как только появится хоть одна реальная метрика — заменить её немедленно (это уже прямо зафиксировано в собственном чек-листе команды — поддерживаем этот приоритет).
- **Локализация** (`i18n.js`, 556 строк) реализована вручную объектами `{ru, en}` без библиотеки — работает, но не поддерживает плюрализацию, форматирование чисел/дат по локали, RTL — при выходе на арабоязычные рынки (в списке валют уже есть AED) потребуется полноценный i18n-фреймворк.

**Что устарело/можно улучшить визуально:**
- Дизайн-токены (`docs/DESIGN_TOKENS.md`, `tokens.css`) существуют — это хорошая практика, редкая на этой стадии. Не увидели явных признаков "конструкторского" вида (нет очевидного использования дефолтных Bootstrap/Material компонентов) — визуально продукт скорее всего выглядит достаточно кастомно. Полноценно оценить эстетику без живого скриншота нельзя — рекомендуем отдельный дизайн-ревью с реальными снимками экрана.

## 7. Логика и потенциальные баги

1. **Баг конвертации TON/BTC** — уже разобран в п.0.3, самый серьёзный.
2. **`reorder_investors`** проверяет, что все `id` существуют, но не проверяет **дубликаты `sortOrder`** — два инвестора с одинаковым порядком отобразятся в непредсказуемом порядке (`ORDER BY sort_order ASC, id ASC` — на практике смягчается вторичной сортировкой по `id`, так что не критично, но неявно).
3. **`mark_latest_pending_paid`** для Stars ищет "последний pending платёж этого пользователя/тарифа" — если пользователь дважды нажмёт "оплатить" на один тариф (двойной клик, см. п.4.10 про Idempotency-Key), создастся два pending-платежа, и вебхук закроет **только один из двух**, второй навсегда останется в статусе `pending` — засоряет историю платежей пользователя недостоверными "неоплаченными" записями.
4. **`consume_oauth_code`** в `repo.py` — функция определена, но при беглом просмотре `auth.py` не вызывается нигде явно (обмен кода идёт через отдельный in-memory `_pending_tokens`, а не через `oauth_links.code`) — похоже на недоделанный/устаревший путь миграции логики, две параллельные системы одноразовых кодов (`oauth_links.code` в БД и `_pending_tokens` в памяти) решают одну и ту же задачу. Нужно оставить одну.
   **✅ ИСПРАВЛЕНО (раунд 3)**: `_pending_tokens` удалён, остался только `oauth_links`/`consume_oauth_code`. Как побочный эффект, сессия/JWT теперь выпускается не на `/callback`, а только в момент реального `/exchange` — раньше при незавершённом флоу (пользователь не вернулся в Telegram) в БД оставалась неиспользуемая "живая" сессия.
5. **`/api/auth/logout`** отзывает сессию только если заголовок начинается с `Bearer ` — для `tma`-сессий (Telegram) логаут не имеет смысла по архитектуре (см. комментарий в коде), но фронтенд должен это явно учитывать в UX ("выйти" внутри Telegram может визуально ничего не поменять, пока initData продолжает валидно генерироваться Telegram-клиентом).
6. **Reorder bulk** не в транзакции с проверкой конкурентного изменения — если два админа одновременно тащат карточки, второй перезапишет порядок первого без предупреждения (lost update). Не критично при 1-2 админах сейчас, важно при росте команды.

## 8. DevOps

Самый слабый раздел проекта на сегодня.

- ❌ **Нет `Dockerfile` и `docker-compose.yml`** — вообще ни одного файла контейнеризации в проекте. Деплой по READMEу описан как "залить в Railway из GitHub" — работает, но не переносимо (vendor lock-in на platform-as-a-service), не воспроизводимо локально в идентичном виде.
  **✅ ИСПРАВЛЕНО (раунд 2)**: добавлены `Dockerfile` (непривилегированный пользователь, healthcheck на `/health`) и `docker-compose.yml` (app + Postgres + Redis для локальной разработки без завязки на живой Supabase).
- ❌ **Нет CI/CD** — ни `.github/workflows`, ни любого другого пайплайна. Значит: нет автоматического запуска линтера/тестов на PR, любой коммит в main может сломать прод, деплой — ручной (`git push` в Railway).
  **✅ ИСПРАВЛЕНО (раунд 2)**: `.github/workflows/ci.yml` — job `backend` (ruff + py_compile + pytest на Postgres-сервисе, с раунда 3 и Redis-сервисе) и job `frontend` (eslint + prettier) на каждый PR/push в `main`.
- ❌ **Нет тестов вообще** — `eslint`/`prettier` настроены на фронте (хорошо), но не запускаются автоматически. `scripts/manual_test_investors_amount.py` и `manual-test-investors.mjs` — это ручные smoke-тесты, не автоматизированные. Backend без единого `pytest`-теста для авторизации, платежей, 2FA — критичных с точки зрения денег и приватности данных путей.
  **✅ ИСПРАВЛЕНО (раунд 2, дополнено в раунде 3)**: 27 тестов на `pytest`+`TestClient` (регистрация/логин, rate-limit и оба его фолбэка, полный цикл 2FA, logout/сессии, чекаут+идемпотентность, HMAC CryptoBot webhook, security-заголовки, `/health`, OAuth-обмен). Гоняются на настоящем Postgres (в проекте нет ORM-слоя, мок на SQLite дал бы ложное покрытие). Тесты сразу нашли реальный баг (`totp_enabled` как Python `int` вместо `bool` в Postgres — 2FA была полностью сломана после первого включения).
- ❌ **Нет мониторинга/алертинга** — нет Sentry/аналога для отслеживания ошибок в проде, нет healthcheck-эндпоинта (`/health`), нет structured logging (сейчас — `print()` в `config.py` для предупреждений, не логгер).
  **✅ ИСПРАВЛЕНО (раунд 2)**: `/health` реально проверяет соединение с БД (`SELECT 1`), не просто "процесс жив". `logging.basicConfig` вместо голых `print()`. Sentry подключается опционально через `SENTRY_DSN` — без переменной поведение не меняется.
- ❌ **Нет staging-окружения** — судя по документации, деплой идёт сразу в прод.
- ⚠️ **База данных на Supabase** — плюс (managed, не привязан к серверу приложения), но нет описанного disaster recovery процесса.
- ⚠️ **Один процесс = сайт + API + аплоады** — нет балансировки, нет readiness/liveness проб, если используется на серверлесс/контейнерном хостинге с авто-рестартом при падении — юзеры увидят downtime без graceful degradation.

Это ожидаемо для сольного/маленького проекта на этой стадии — но именно это первым проверит любой серьёзный технический due diligence перед чеком в $100M, и именно здесь разрыв между "работает у меня" и "институциональный продакшн" самый большой.
