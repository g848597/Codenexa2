## Раунд 5

Тема раунда: раздел 13 аудита, "🟡 Средний приоритет" — 6 независимых задач,
решаются по одной в порядке из отчёта. Этот раунд закрывает первую.

### 1. Пул соединений к Postgres (`db.py`)

**Было**: `db.py` держал одно psycopg2-соединение на ОС-поток через
`threading.local` бессрочно, переоткрывая только при обрыве. FastAPI гоняет
sync-эндпоинты через threadpool (до 40 потоков по умолчанию) — то есть до 40
постоянно открытых соединений с одного процесса, и это число растёт с каждым
дополнительным воркером. Supabase Session Pooler ограничивает число
соединений (обычно 15-60 в зависимости от плана) — при реальной нагрузке
это грозило `too many connections`.

**Стало**: соединения берутся из `psycopg2.pool.ThreadedConnectionPool`
(`app/web/db.py`) на время ОДНОЙ единицы работы:
- `get_conn()` — на время одного `execute()`, строки вычитываются заранее и
  отдаются через лёгкий `_EagerCursor`, чтобы `.fetchone()`/`.fetchall()`
  работали как раньше уже после того, как реальное соединение вернулось в
  пул;
- `tx()` — на время всего `with`-блока транзакции (нужно для атомарности
  многошаговых операций), возвращается в пул сразу по выходу из блока (успех
  или исключение — не важно).

Размер пула настраивается через `DB_POOL_MIN`/`DB_POOL_MAX` в `.env`
(по умолчанию 1/10 на процесс). При деплое с несколькими воркерами суммарное
число соединений = воркеры × `DB_POOL_MAX` — держите это в пределах лимита
плана Supabase (см. README_BACKEND.md, раздел 7, пункт 6).

**Почему НЕ через thread-local + middleware**: первый вариант реализации
кэшировал соединение в `threading.local` и возвращал его в пул из
middleware в `finally`-блоке после ответа. Оказалось, что это ненадёжно:
sync-эндпоинты FastAPI выполняются в threadpool (реальный отдельный ОС-поток
через `anyio.to_thread`), а middleware и async-эндпоинты — в потоке event
loop. Для одного и того же HTTP-запроса это могут быть РАЗНЫЕ ОС-потоки на
входе (где эндпоинт открывает соединение) и на выходе (где middleware
пытается его освободить) — thread-local-привязка в такой схеме периодически
"теряла" бы соединение: оно бы оставалось занятым до случайного повторного
попадания на тот же поток, а не возвращалось в пул сразу. Итоговая схема
("взять из пула — использовать — сразу вернуть", целиком в одном вызове
`get_conn()`/`tx()`) свободна от этой проблемы, т.к. не пересекает границу
поток/event-loop.

**Не входило в scope этого пункта** (отдельная, более крупная задача из
раздела 3 аудита, не из раздела 13): синхронный psycopg2 внутри `async def`
эндпоинтов (`google_callback`, `yandex_callback`, `cryptobot_webhook`,
`telegram_webhook`) по-прежнему блокирует event loop на время запроса к БД.
Пул решает проблему количества соединений, но не проблему блокировки —
это кандидат на будущий раунд (обернуть в `run_in_threadpool` или сделать
эндпоинты `def`).

**Файлы**: `app/web/db.py` (основная логика), `app/web/config.py`
(`DB_POOL_MIN`/`DB_POOL_MAX`), `README_BACKEND.md` (раздел 7, документация).
Публичный интерфейс `get_conn()`/`tx()`, которым пользуется `repo.py`, не
менялся — правки полностью внутренние.

**Ограничение проверки**: в этом окружении нет доступа к сети/локальному
Postgres, поэтому изменения проверены статически (синтаксис, построчный
разбор всех мест использования `get_conn()`/`tx()`/`executemany()` в
`repo.py` и `api/`) и логическим review, но не прогонкой `pytest` против
реальной БД (`tests/conftest.py` требует `TEST_DATABASE_URL`). Перед мёрджем
рекомендуется прогнать `pytest` локально/в CI, где Postgres доступен —
workflow `.github/workflows/ci.yml` это уже делает.

### 2. `payments.amount`: TEXT → NUMERIC + единая денежная утилита

**Было**: `payments.amount` хранился как `TEXT`. Суммы гонялись по коду как
строки/`float` в разных представлениях — цена тарифа в `PLANS` (строка),
конвертация в крипту через `float`-деление в `cryptobot.convert_usd_to_asset()`,
курс с CryptoBot API тоже `float`. Деньги через `float` — источник тихой
потери точности (IEEE754 не хранит десятичные дроби точно), а `TEXT` в
колонке `amount` ломает сортировку/агрегацию на уровне SQL ("100.00" <
"99.9" как строки — неверно).

**Стало**:
- `payments.amount` — `NUMERIC(20, 8)` (миграция для существующих БД: ALTER
  COLUMN ... USING amount::NUMERIC, идемпотентно, см. `_run_migrations()` в
  `db.py`);
- новый модуль `app/web/money.py` — единая точка работы с деньгами:
  `to_decimal()` безопасно приводит `str`/`int`/`float`/`Decimal` к `Decimal`
  (через `repr()` для `float`, чтобы не утащить двоичную погрешность),
  `quantize()`/`to_display()` форматируют сумму под конкретную валюту
  (2 знака для USD/USDT, 4 для TON, 8 для BTC, 0 для XTR/Stars), округление
  всегда вниз;
- `cryptobot.get_usd_rate()`/`convert_usd_to_asset()` переведены с `float` на
  `Decimal`;
- `repo.create_payment()` приводит `amount` к `Decimal` перед записью
  независимо от того, что передал вызывающий код;
- в API (`billing.py`) суммы в JSON-ответах отдаются форматированной строкой
  через `money.to_display()`, а не "голым" `Decimal` — иначе сериализация по
  умолчанию (обычно через `float`) вернула бы ту же самую погрешность,
  ради ухода от которой всё это делалось.

**Файлы**: `app/web/money.py` (новый), `app/web/db.py` (схема + миграция),
`app/web/repo.py::create_payment`, `app/web/api/billing.py`,
`app/web/integrations/cryptobot.py`.

**Тесты**: `tests/test_money_and_pool.py` — юнит-тесты `money.py` + round-trip
тест `payments.amount` через `repo.create_payment`/`list_payments` + тест на
много последовательных запросов подряд.

## Раунд 5.1 — верификация против реального Postgres, 2 найденных и
исправленных бага

При составлении раунда 5 не было доступа к Postgres, поэтому пул и миграция
`payments.amount` были проверены только статически (см. пометку "Ограничение
проверки" в разделе выше). В этом раунде поднят реальный Postgres 16 и весь
`pytest` прогнан по-настоящему — обнаружилось, что после раунда 5 **бэкенд
был полностью нерабочим**: буквально ни один запрос к БД не проходил.
Оба бага — прямое следствие миграции на пул соединений (п.1 раунда 5), их
не было до раунда 5.

### Баг 1: `set_session cannot be used inside a transaction` на КАЖДОМ запросе

**Было** (`_borrow_conn()` в `db.py`): сразу после `pool.getconn()` шёл
пробный `SELECT 1` (проверка, что соединение живое), и только ПОСЛЕ него —
`conn.autocommit = True`. Но сам пробный `SELECT 1` уже открывал неявную
транзакцию (psycopg2 по умолчанию `autocommit=False` на новом соединении), а
psycopg2/libpq запрещают менять `autocommit` внутри открытой транзакции —
любой вызов падал с `psycopg2.ProgrammingError: set_session cannot be used
inside a transaction`. Это ловилось только при реальном обращении к БД,
поэтому в раунде 5 (без Postgres под рукой) осталось незамеченным.

**Стало**: `conn.autocommit = True` выставляется СРАЗУ после получения
соединения из пула, до пробного запроса (с обработкой редкого случая, если
конкретное соединение всё же вернулось в пул с незакрытой транзакцией —
тогда `rollback()` и повтор).

### Баг 2: `create_user()`/`create_investor()` возвращали `None` после INSERT

**Было**: обе функции внутри `with tx() as conn:` делали `INSERT`, а затем
читали только что вставленную строку через `get_user_by_id()`/
`get_investor()` — но те используют `get_conn()`, который берёт из пула
**другое** соединение. Транзакция ещё не закоммичена (`commit()` происходит
только по выходу из `with tx()`), поэтому по правилам изоляции Postgres
второе соединение не видело вставленную строку и `fetchone()` возвращал
`None` → `TypeError: 'NoneType' object is not subscriptable` при обращении к
`user["id"]` в вызывающем коде (например, `/api/auth/register`). До
миграции на пул это работало случайно: один физический коннекшен на поток
означал, что "другое" соединение фактически было тем же самым.

**Стало**: обе функции читают вставленную строку через тот же `conn`,
которым делали `INSERT`, а не через отдельный `get_conn()`.

**Проверено**: полный `pytest` (52 теста) на реальном Postgres 16 —
49 passed, 3 skipped (требуют Redis, не поднимался в этом раунде). Все
тесты, включая `test_money_and_pool.py` и `test_oauth_exchange.py`, ранее
не прогонявшиеся вживую, теперь зелёные.

**Файлы**: `app/web/db.py` (`_borrow_conn`), `app/web/repo.py`
(`create_user`, `create_investor`).

## Раунд 6

Тема раунда: раздел 13 аудита, "🟡 Средний приоритет" — следующая по списку
задача, "Ролевая модель админов (вместо allow-list)".

### Ролевая модель админов вместо allow-list

**Было**: `is_admin_user()` (`app/web/deps.py`) сверял email/telegram_id
текущего пользователя со статичными множествами `ADMIN_EMAILS`/
`ADMIN_TELEGRAM_IDS`, прочитанными из `.env` при старте процесса. Работало
для 1-2 администраторов, но: выдача или отзыв доступа требовали правки
`.env` и передеплоя/рестарта; не было различия между "может модерировать
раздел Инвесторы" и "может назначать других админов" — любой в списке имел
одинаково полный доступ; отозвать скомпрометированный доступ можно было
только редактированием `.env` на хостинге.

**Стало**:
- `users.role` — новая колонка (`TEXT NOT NULL DEFAULT 'user' CHECK (role IN
  ('user', 'admin', 'superadmin'))`), миграция для существующих БД —
  идемпотентно, см. `_run_migrations()` в `db.py`;
- `admin` — доступ к существующим admin-эндпоинтам (сейчас только раздел
  "Инвесторы", `/api/investors/admin/*`), как раньше;
- `superadmin` — дополнительно может выдавать/отзывать роли другим
  пользователям через новый `/api/admin/users`;
- `app/web/deps.py`: `is_admin_user()` и новая `is_superadmin_user()`
  проверяют `role` из БД вместо множеств из `.env`; новая зависимость
  `get_current_superadmin` — по аналогии с `get_current_admin`, но требует
  именно `superadmin`;
- новый роутер `app/web/api/admin_users.py` (только `superadmin`):
  `GET /api/admin/users` (без `?q=` — список текущих admin/superadmin, с
  `?q=` — поиск по email/telegram_id), `PUT /api/admin/users/{id}/role`;
- защита от самоблокировки: нельзя понизить роль самому себе и нельзя
  понизить последнего `superadmin` в системе — обе проверки на сервере,
  не только в UI.

**Проблема холодного старта и её решение**: чтобы выдать роль через
`/api/admin/users`, там уже нужен хотя бы один `superadmin` — курица и яйцо
для самой первой инсталляции. `ADMIN_EMAILS`/`ADMIN_TELEGRAM_IDS` в `.env`
никуда не делись, но сменили смысл на **bootstrap-only**:
`deps._apply_admin_bootstrap()` при каждом аутентифицированном запросе
проверяет, что (а) пользователь — в одном из списков `.env` и (б) в БД **нет
вообще ни одного `superadmin`** — и если оба условия верны, выдаёт роль
`superadmin` автоматически. Как только в системе появился хотя бы один
`superadmin` (в т.ч. через этот же bootstrap), условие (б) больше никогда не
выполняется — значит `.env` больше не может ни выдать, ни (что важнее)
незаметно восстановить роль в обход явного отзыва другим `superadmin` через
API. Проверка "нет ни одного `superadmin` в БД" была осознанным выбором
именно по этой причине — альтернатива "роль ещё не назначена этому
пользователю" обесценивала бы отзыв доступа (следующий логин тут же вернул
бы роль обратно, пока запись остаётся в `.env`).

Bootstrap встроен в единую точку разрешения пользователя
(`deps.try_get_current_user_value`), поэтому срабатывает одинаково что для
`Authorization: tma ...` (Telegram), что для `Authorization: Bearer ...`
(email/Google/Яндекс) — и продублирован внутри `_public_user()`
(`app/web/api/auth.py`), чтобы поле `isAdmin` было верным уже в ответе
`/api/auth/register`/`/login`/`/telegram`/`/exchange`, а не только начиная
со следующего вызова `/api/auth/me`.

**Найденный при верификации баг**: изначальная версия миграции добавляла
`CREATE INDEX ... ON users(role) ...` сразу в блок `SCHEMA` (выполняется на
каждом старте через `CREATE TABLE IF NOT EXISTS`). На **новой** БД это
работало, но на БД, где таблица `users` уже существовала без колонки `role`
(ровно тот случай, для которого и нужна миграция), `CREATE TABLE IF NOT
EXISTS` не трогает существующую таблицу — и `CREATE INDEX` на
ещё-не-добавленную колонку падал с `UndefinedColumn` до того, как код миграции
ниже успевал её добавить. Найдено вручную (не через `pytest`, у которого
свежая тестовая БД и баг не проявляется) — накатил схему, руками удалил
колонку/constraint/индекс, повторно прогнал `init_db()`. Индекс перенесён
из `SCHEMA` в `_run_migrations()`, после блока добавления колонки — тот же
паттерн, что уже использовался для `idx_payments_user_idempotency` в
раунде 5. Проверено повторно тем же ручным сценарием — миграция теперь
идемпотентна и на "старой" схеме.

**Не входило в scope этого пункта**: аудит-лог админ-действий (кто, когда и
что изменил через `/api/admin/users` или `/api/investors/admin/*`) — это
отдельный пункт раздела 13 аудита, следующий по списку.

**Файлы**: `app/web/db.py` (схема + миграция), `app/web/repo.py` (функции
работы с ролями), `app/web/deps.py` (bootstrap + проверки ролей),
`app/web/api/admin_users.py` (новый), `app/web/api/auth.py` (`_public_user`),
`app/web/api/investors.py` (обновлён комментарий), `app/web/server.py`
(регистрация роутера), `app/web/config.py` (обновлён комментарий про
bootstrap-семантику `ADMIN_EMAILS`/`ADMIN_TELEGRAM_IDS`), `README_BACKEND.md`

**Тесты**: `tests/test_admin_roles.py` (14 тестов) — bootstrap на холодном
старте, отсутствие повторного bootstrap после того, как в системе уже есть

выданная через API роль `admin` действительно открывает существующий
`/api/investors/admin`. Полный `pytest` — 63 passed, 3 skipped (Redis) на
реальном Postgres 16.
