# CodeNexa — бэкенд авторизации и оплаты

Что добавлено поверх существующего фронтенда (`webapp/`):

- **`app/web/`** — бэкенд на FastAPI + PostgreSQL (Supabase). Раздаёт и API
  (`/api/...`), и сам фронтенд (`webapp/`), с одного процесса и домена — CORS
  не нужен.
- Полноценная авторизация: **Telegram** (тихий вход внутри мини-аппа), **Email +
  пароль**, **Google**, **Яндекс**.
- Личный кабинет во фронтенде: профиль, привязанные способы входа, смена
  пароля, **2FA (TOTP)**, список активных сессий с отзывом, тарифы и история
  платежей.
- Приём оплаты: **CryptoBot (Crypto Pay API)** и **Telegram Stars**.

Всё протестировано локально (регистрация/логин/2FA/сессии, проверка подписи
Telegram initData) перед сборкой.

## 1. Установка и первый запуск (локально)

```bash
cd codenexa
python3 -m venv .venv && source .venv/bin/activate      # или без venv, если не принципиально
pip install -r requirements.txt

# .env уже создан и заполнен вашими токенами (бот + CryptoBot).
# Откройте его и проверьте/дозаполните значения — см. раздел 2 и 3 ниже.

uvicorn app.web.server:app --reload --port 8000
```

Откройте `http://localhost:8000` — увидите тот же мини-апп, а по адресу
`http://localhost:8000/docs` — автосгенерированную документацию API (Swagger).

**Важно про токен бота**: он был отправлен в открытом чате, поэтому по-хорошему
стоит перевыпустить его в @BotFather (`/mybots` → выбрать бота → `API Token` →
`Revoke`) и вписать новый в `.env`.

## 2. Настройка Google OAuth

1. Откройте https://console.cloud.google.com/apis/credentials
2. Создайте проект (если ещё нет) → `Create Credentials` → `OAuth client ID`
3. Тип приложения: `Web application`
4. `Authorized redirect URIs` → добавьте:
   - для локальной разработки: `http://localhost:8000/api/auth/google/callback`
   - для продакшена: `https://ВАШ_ДОМЕН/api/auth/google/callback`
5. Скопируйте `Client ID` и `Client secret` в `.env`:
   ```
   GOOGLE_CLIENT_ID=...
   GOOGLE_CLIENT_SECRET=...
   GOOGLE_REDIRECT_URI=https://ВАШ_ДОМЕН/api/auth/google/callback
   ```
6. На экране `OAuth consent screen` укажите название приложения и логотип —
   иначе пользователи увидят предупреждение "непроверенное приложение".

## 3. Настройка Яндекс OAuth

1. Откройте https://oauth.yandex.ru/client/new
2. Платформа: `Веб-сервисы`, Redirect URI:
   `https://ВАШ_ДОМЕН/api/auth/yandex/callback`
3. Права доступа: `Доступ к email`, `Доступ к портрету`, `Доступ к имени и фамилии`
4. Скопируйте `ID` и `Пароль` в `.env`:
   ```
   YANDEX_CLIENT_ID=...
   YANDEX_CLIENT_SECRET=...
   YANDEX_REDIRECT_URI=https://ВАШ_ДОМЕН/api/auth/yandex/callback
   ```

Пока не настроите — кнопки Google/Яндекс в карточке входа будут показывать
понятную ошибку, остальное (Telegram, email/пароль) работает уже сейчас.

## 4. Как работает вход через Google/Яндекс внутри Telegram

Telegram Mini App открыт во встроенном браузере (WebView), где сторонние
OAuth-редиректы иногда работают нестабильно. Поэтому:

1. Пользователь жмёт "Google"/"Яндекс" → мини-апп открывает ссылку через
   `Telegram.WebApp.openLink()` — это **системный браузер**, а не WebView.
2. После успешного входа сервер показывает страницу "Вернитесь в Telegram" со
   ссылкой `t.me/<бот>?start=auth_<код>`.
3. Пользователь возвращается в Telegram → бот открывает мини-апп с этим
   параметром → мини-апп сам обменивает код на токен (`/api/auth/exchange`) и
   человек уже залогинен.

Это стандартный обходной путь для OAuth в мини-аппах, ничего вручную делать не
нужно — только правильно указать `TELEGRAM_BOT_USERNAME` в `.env`.

## 5. Вебхук самого бота (нужен для Telegram Stars)

После деплоя на публичный домен выполните один раз:

```bash
curl "https://api.telegram.org/bot<TELEGRAM_BOT_TOKEN>/setWebhook?url=https://ВАШ_ДОМЕН/api/telegram/webhook/<TELEGRAM_WEBHOOK_SECRET>"
```

Значения `TELEGRAM_BOT_TOKEN` и `TELEGRAM_WEBHOOK_SECRET` — из вашего `.env`.
Без этого шага оплата Stars будет создавать счёт, но бэкенд не узнает об
успешной оплате.

## 6. Вебхук CryptoBot (необязательно, но рекомендуется)

В @CryptoBot → `Crypto Pay` → ваше приложение → `Webhooks` → укажите:
`https://ВАШ_ДОМЕН/api/billing/cryptobot/webhook`

Без вебхука платежи всё равно создаются, но статус "оплачено" не будет
подтверждён автоматически (см. TODO по проверке подписи в
`app/web/api/billing.py`, если понадобится — допишу).

## 7. Роли администраторов

Доступ к админ-панелям (сейчас: `/api/investors/admin/*`, раздел "Инвесторы")
завязан на роль в БД (`users.role`: `user` | `admin` | `superadmin`), а не на
статичный список в `.env` — так можно выдавать/отзывать доступ через API,
без правки `.env` и передеплоя.

**Холодный старт** (первый администратор в новой инсталляции): пропишите свой
email и/или Telegram ID в `.env`:
```
ADMIN_EMAILS=you@example.com
ADMIN_TELEGRAM_IDS=123456789
```
Затем один раз войдите под этим email/аккаунтом (email/пароль, Google, Яндекс
или Telegram — любой способ входа, привязанный к этому email/ID) — при первом
же авторизованном запросе вам автоматически будет выдана роль `superadmin`.
Это срабатывает **только пока в БД нет вообще ни одного superadmin** — после
этого `ADMIN_EMAILS`/`ADMIN_TELEGRAM_IDS` в `.env` больше ни на что не влияют,
дальше управление ролями идёт исключительно через API ниже (подробности и
обоснование — в комментариях `app/web/deps.py::_apply_admin_bootstrap`).

**Управление ролями** (нужна роль `superadmin`):
- `GET /api/admin/users` — список текущих admin/superadmin.
- `GET /api/admin/users?q=часть_email_или_telegram_id` — поиск пользователя,
  которому нужно выдать роль.
- `PUT /api/admin/users/{id}/role` — тело `{"role": "user"|"admin"|"superadmin"}`.

Ограничения (проверяются на сервере, не только в UI):
- Обычный `admin` не может ни смотреть, ни менять роли — только `superadmin`.
- Нельзя понизить роль самому себе (защита от случайной потери доступа).
- Нельзя понизить последнего `superadmin` в системе — всегда должен
  оставаться хотя бы один, иначе восстановить доступ можно только прямым
  вмешательством в БД.

## 8. Деплой

База теперь на Supabase (Postgres) — она не привязана к серверу, поэтому
сервер, на котором крутится сам бэкенд, можно менять когда угодно: данные
переезжать не нужно, только скопировать код + `.env` с тем же `DATABASE_URL`.

Рекомендация для старта — **Railway** (railway.com): деплой из GitHub одной
кнопкой, бесплатный пробный период без карты, дальше от ~$5/мес. Альтернативы:
Render.com, собственный VPS (тогда — `gunicorn`/`uvicorn` за nginx,
systemd-юнит) — постоянный диск для БД больше не нужен ни в каком варианте.

Общий план для любого варианта:
1. Залить репозиторий (без `.env` — он в `.gitignore`).
2. В настройках хостинга вставить переменные окружения из вашего `.env`
   (это делается через панель хостинга, не через файл) — включая
   `DATABASE_URL`.
3. Указать команду запуска: `uvicorn app.web.server:app --host 0.0.0.0 --port $PORT`
4. Прописать реальный `PUBLIC_BASE_URL` и обновить `GOOGLE_REDIRECT_URI` /
   `YANDEX_REDIRECT_URI` — и в `.env`, и в консолях Google/Яндекс.
5. Выполнить шаг 5 из раздела 5 выше (setWebhook) с новым доменом.
6. (опционально, но рекомендуется при переходе с одного воркера на несколько)
   Проверить `DB_POOL_MAX` (по умолчанию 10) — это верхняя граница
   одновременно занятых соединений к Postgres **на один процесс/воркер**.
   Умножьте на число воркеров/инстансов и сравните с лимитом соединений
   вашего плана Supabase (Settings → Database → Connection pooling), чтобы
   не словить `too many connections` при росте нагрузки.

## 8.1 Staging-окружение

Раунд 8 (аудит, раздел 9): `ENV` поддерживает третье значение — `staging`,
не только `development`/`production`. По требованиям безопасности staging
ведёт себя **так же, как production** (secure-cookie, HSTS, обязательный
`JWT_SECRET` при старте — см. `app/web/config.py::Settings.is_production_like()`),
потому что это реальный HTTPS-доступный деплой, а не локальная разработка.

Минимальная настройка staging на Railway (или аналоге):
1. Создайте второй сервис в том же или отдельном Railway-проекте — не
   переиспользуйте production-сервис.
2. Отдельная база: либо второй Supabase-проект, либо отдельная схема/база в
   том же — не пускайте staging писать в те же таблицы, что и production.
3. Переменные окружения — те же, что в разделе 8, плюс `ENV=staging`.
   `JWT_SECRET` должен быть **другим**, не production-секретом (иначе
   staging-токены будут приниматься production-сервером и наоборот).
4. Свой `PUBLIC_BASE_URL`, свои `GOOGLE_REDIRECT_URI`/`YANDEX_REDIRECT_URI`
   (заведите отдельные OAuth-клиенты в консолях Google/Яндекс — тестовые
   redirect_uri в проде создавать не нужно) и свой `TELEGRAM_WEBHOOK_SECRET`
   (иначе тестовые вебхуки Stars будут дёргать production-бота).

## 8.2 Backup / восстановление (Supabase)

База — Supabase Postgres, поэтому backup/restore идёт через саму Supabase, а
не через код этого репозитория:

- **Point-in-Time Recovery (PITR)** — Settings → Database → Backups в
  Supabase-проекте. На платных планах доступно восстановление на
  произвольный момент времени за последние N дней (зависит от плана) — это
  основной механизм восстановления после случайного `DELETE`/`DROP` или
  повреждения данных.
- **Ручной снапшот перед рискованными операциями** (миграция схемы вручную,
  массовое исправление данных) — `pg_dump` через `DATABASE_URL`:
  ```
  pg_dump "$DATABASE_URL" -Fc -f backup_$(date +%Y%m%d_%H%M).dump
  ```
  Восстановление: `pg_restore -d "$DATABASE_URL" backup_YYYYMMDD_HHMM.dump`
  (на **отдельную** тестовую базу для проверки, не поверх продакшна вслепую).
- **RPO/RTO — зафиксируйте явно, когда выберете план Supabase**: PITR-планы
  обычно дают RPO в единицы секунд (WAL-репликация) и RTO — время, нужное
  чтобы поднять восстановленный инстанс и переключить `DATABASE_URL`
  (обычно минуты, не часы). На бесплатном плане Supabase PITR недоступен —
  единственная страховка там — ручные `pg_dump`-снапшоты по расписанию
  (например, через cron/GitHub Actions), с RPO, равным интервалу между
  снапшотами.
- Это единственный технический долг из аудита (раздел 9), который
  сознательно не закрывается кодом — сам процесс должен быть решением
  владельца (какой план Supabase, как часто снимать ручные снапшоты), не
  чем-то, что можно захардкодить в репозитории.

## 9. Список тарифов

Раунд 8 (аудит, раздел 12 "Объединить"): тарифы больше не статичный
Python-словарь — они в таблице `plans` (см. `app/web/db.py`), с полной
историей изменения цен. При первом запуске (`init_db()`) таблица сама
заполняется теми же значениями-плейсхолдерами, что были раньше
(`Pro — месяц` / `Pro — год`) — замените на реальные тарифы и цены через:

```
PUT /api/admin/plans/{code}
Authorization: Bearer <токен superadmin>
Content-Type: application/json

{"title": "Pro — месяц", "usd": "9.00", "stars": 500}
```

Старая цена не удаляется, а остаётся в истории — `GET /api/admin/plans/history?code=pro_monthly`
покажет все изменения. Публичный список действующих тарифов — `GET /api/billing/plans`
(без авторизации), административный — `GET /api/admin/plans` (только superadmin).

### Реферальная программа

`REFERRAL_REWARD_USD` (`.env`, необязательно) — сумма вознаграждения
рефереру за первую оплату приглашённого. Пока не задана — программа честно
ведёт учёт (кто кого пригласил, у кого была первая оплата), но не
начисляет выдуманную сумму (см. `app/web/referrals.py`). Код реферальной
ссылки — `telegram_id` пригласившего (`t.me/<бот>?start=ref_<telegram_id>`).
