# Раунд 7 — Аудит-лог админ-действий

Взят следующий пункт из дорожной карты `CodeNexa_Audit_100M.md` (раздел 13,
"Средний приоритет"): **аудит-лог админ-действий**, ранее отмеченный как
"ещё не начато".

## Что сделано

1. **Новая таблица `admin_audit_log`** (`app/web/db.py`): `admin_id`, `action`,
   `target_type`, `target_id`, `details` (JSONB), `ip`, `created_at`. Индексы
   на `created_at`, `admin_id`, `(target_type, target_id)`. Только
   INSERT/SELECT в коде — нет ни одного эндпоинта для изменения/удаления
   записей: аудит-лог, который можно редактировать, не аудит-лог.

2. **`app/web/audit.py`** — общая точка записи (`log_action()`), чтобы не
   дублировать try/except-обвязку в каждом `api/*.py` (тот же принцип, что и
   вынос `esc()/escAttr()` в `webapp/src/utils/html.js` в прошлых раундах).
   Запись — best-effort: сбой логирования никогда не откатывает и не роняет
   уже выполненное и закоммиченное админ-действие.

3. **Что логируется:**
   - `role_change` — смена роли пользователя (`admin_users.py`), с `from`/`to`
     и email цели в `details`.
   - `create` / `update` / `delete` / `reorder` / `photo_upload` /
     `photo_delete` — все изменяющие действия над карточками инвесторов
     (`investors.py`).

4. **`GET /api/admin/users/audit-log`** — только `superadmin` (сам факт "кто
   что менял" — чувствительные данные, не для обычного `admin`). Параметры:
   `limit` (≤200), `offset`, `action`, `adminId`. Отдаёт email/имя админа
   через `LEFT JOIN` на `users`, отсортировано по времени убывания.

5. **Тесты** (`tests/test_admin_audit_log.py`, 5 новых): запись при смене
   роли, запись при CRUD инвесторов (порядок новые→старые), 403 для
   обычного admin и анонима, фильтрация по `action`.

  16, без regressions в существующих тестах).
- `ruff check` — все изменённые файлы чисты (0 замечаний).
